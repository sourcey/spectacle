# Distribution Blowup Plan

> Get sourcey installed everywhere docs are built. Every generated site is a backlink. Every registry listing is a permanent link to sourcey.com.

---

## How Sourcey Differs from Icey

Icey's play is distro package managers and mirrors. Sourcey's play is different:

1. **Generated output** — every site built with sourcey can carry a "Built with Sourcey" link. This is how Hugo, Docusaurus, and MkDocs built massive backlink profiles. One install can produce thousands of pages, each a potential backlink
2. **Developer tooling registries** — npm is the primary channel, but there are aggregators and directories that multiply visibility
3. **Template/starter ecosystem** — starter repos get forked. Forks carry your links
4. **Integration surface** — CI actions, IDE extensions, and platform integrations each create a new registry listing

The volume ceiling is higher than icey because the output multiplies.

## Tier 1: Generated Output as Backlink Engine

This is the highest-ROI move. Every site sourcey generates becomes a backlink source.

### 1. Default "Built with Sourcey" Footer

Every generated docs site should include a subtle footer link by default:

- "Built with [Sourcey](https://sourcey.com)" in the default theme footer
- HTML meta generator tag: `<meta name="generator" content="Sourcey 3.3.8">`
- both are removable by users, but most won't bother (same as Hugo, Jekyll, WordPress)
- the meta tag is invisible but indexed by search engines and tracked by aggregators like BuiltWith and Wappalyzer

This is how WordPress powers 40% of the web and everyone knows it. The generator tag alone.

### 2. Default OpenGraph / Social Tags

When generated sites are shared on social media:

- include `og:site_name` or a subtle attribution in default social cards
- not intrusive, but creates association when docs are shared

### 3. Sitemap Attribution

Generated sitemaps can include a comment: `<!-- Generated by Sourcey https://sourcey.com -->`

Sitemaps are submitted to Google Search Console and crawled by every search engine.

## Tier 2: Registry and Directory Listings

### 4. npm (already live)

Optimise the existing listing:

- ensure `homepage` points to `sourcey.com` (currently points to GitHub readme)
- add `funding` field in package.json pointing to sourcey.com or a sponsor page
- keywords are good but could add: `markdown`, `developer-documentation`, `api-docs`, `swagger-docs`

### 5. Docker Hub / GitHub Container Registry

Ship an official Docker image:

- `docker run sourcey/sourcey build` — zero-install docs generation
- Docker Hub pages are highly indexed: hub.docker.com/r/sourcey/sourcey
- every `FROM sourcey/sourcey` in a downstream Dockerfile is a reference
- CI pipelines commonly use Docker images, creating references in public workflow files

### 6. GitHub Actions Marketplace

Create `sourcey/build-docs` action:

- generates and optionally deploys docs in CI
- GitHub Actions marketplace listing is a permanent, indexed page
- every `uses: sourcey/build-docs@v1` in a public workflow is a backlink
- this is how Vercel and Netlify bootstrapped CI-driven adoption

### 7. VS Code Extension Marketplace

Ship a sourcey VS Code extension:

- live preview of sourcey docs while editing
- marketplace.visualstudio.com listing — massive traffic, indexed
- every install shows up in extension counts, building social proof

### 8. Homebrew

Unusual for a node CLI but not unheard of (vercel, netlify-cli, wrangler):

- `brew install sourcey` removes the node/npm prerequisite for non-JS users
- formulae.brew.sh listing is another indexed backlink
- start with a tap (`nilstate/homebrew-sourcey`), promote to homebrew-core later

### 9. Nix

- add a `flake.nix` to the sourcey repo
- submit to nixpkgs
- same mirror/aggregator multiplier as icey

## Tier 3: Ecosystem and Integration Surface

### 10. Starter Templates

Create `sourcey/sourcey-starter` and topic-specific starters:

- `sourcey-starter-openapi` — for API docs
- `sourcey-starter-guides` — for product guides
- every fork retains README links, package.json homepage, and theme attribution
- GitHub template repos get a "Use this template" button — high conversion

### 11. Plugin / Theme Ecosystem

If sourcey supports custom themes:

- ship 2-3 official themes as separate npm packages
- each is a separate npm listing with `homepage: sourcey.com`
- community themes naturally link back to the parent project
- theme showcase page on sourcey.com creates a reason for theme authors to link to you

### 12. Integration Docs on Other Platforms

Get sourcey mentioned in:

- GitHub Pages docs (as a supported generator)
- Netlify/Vercel/Cloudflare Pages deploy guides
- CI platform docs (GitHub Actions, GitLab CI examples)
- each mention is a high-authority backlink from a trusted domain

### 13. Awesome Lists and Directories

Submit sourcey to:

- `awesome-openapi` (GitHub awesome list)
- `awesome-static-generators`
- `awesome-documentation-tools`
- `staticgen.com` / `jamstack.org/generators`
- `openapi.tools` directory
- `alternativeto.com`
- each is a curated, indexed, high-authority backlink

## Tier 4: Aggregator and Tracking Services

These happen automatically once the generator tag and npm package are live:

### 14. BuiltWith

BuiltWith tracks technologies via meta generator tags and script signatures. Once sourcey sites exist with the generator tag, BuiltWith auto-catalogs them. This creates a technology profile page for sourcey.

### 15. Wappalyzer

Same principle. Wappalyzer detects technologies and maintains a public directory. A Wappalyzer detection rule can be submitted for sourcey.

### 16. npm Trends / Bundlephobia / Socket.dev

These sites auto-index every npm package. Optimize for them:

- keep bundle size reasonable (bundlephobia.com)
- maintain clean dependency tree (socket.dev)
- npm-trends.com comparison pages drive discovery

## Backlink Mechanics

| Source | How | Volume |
|---|---|---|
| Generated docs footer | "Built with Sourcey" on every page | 1 per page per site (potentially thousands per user) |
| Generator meta tag | `<meta name="generator">` indexed by crawlers | 1 per site, tracked by BuiltWith/Wappalyzer |
| npm | package page with homepage link | 1, permanent |
| Docker Hub | image page | 1, permanent |
| GitHub Actions marketplace | action listing | 1 + every workflow reference |
| VS Code marketplace | extension listing | 1, high traffic |
| Starter repo forks | README, package.json links carried through | grows with adoption |
| Awesome lists / directories | curated links | 5-10 high-authority |
| BuiltWith / Wappalyzer | auto-detected technology profiles | 1 per aggregator |
| Homebrew / Nix | formula pages and mirrors | same multiplier as icey |

### Metadata Consistency

Every surface must point to `sourcey.com`:

- `package.json` → `homepage`
- Docker image labels → `org.opencontainers.image.url`
- GitHub Action → `branding` + metadata
- VS Code extension → `package.json` homepage
- Generated HTML → meta generator tag
- Homebrew formula → `homepage`
- Nix derivation → `meta.homepage`
